### Session 2026-07-13

- Q: 추천 대상은 주변 식당의 실제 메뉴인가, 일반 메뉴 아이디어인가? → A: 주변 식당 메뉴와 일반 메뉴 아이디어를 함께 제공한다.
- Q: 실제 메뉴와 일반 메뉴 아이디어를 어떻게 표시할 것인가? → A: 사용자가 추천 유형을 먼저 선택한다.
- Q: 실제 식당 추천에 위치를 어떻게 사용할 것인가? → A: 위치 권한을 요청하고, 거부하면 주소나 동네를 직접 입력하게 한다.
- Q: 실제 식당 정보가 최신인지 확인할 수 없을 때 어떻게 처리할 것인가? → A: 최신 정보가 확인된 항목만 추천하고 확인 불가 항목은 제외한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 오늘의 점심 추천 받기 (Priority: P1)

사용자는 점심 메뉴를 고르는 데 시간을 쓰지 않고, 주변 식당의 실제 메뉴 또는
위치와 무관한 일반 메뉴 아이디어 중 추천 유형을 선택해 추천받는다.

**Why this priority**: 메뉴 선택 문제를 직접 해결하는 핵심 가치이므로 가장
먼저 제공해야 한다.

**Independent Test**: 사용자가 서비스에 들어와 추천 유형을 선택하고 요청하면
선택한 유형의 점심 메뉴 또는 메뉴 아이디어와 각 항목의 간단한 이유를 확인할
수 있다.

**Acceptance Scenarios**:

1. **Given** 사용자가 추천 유형을 선택한 상태에서, **When** 점심 추천을
   요청하면, **Then** 서비스는 선택한 유형의 결과를 하나 이상 보여준다.
2. **Given** 추천 결과가 표시된 상태에서, **When** 사용자가 결과를 확인하면,
   **Then** 각 메뉴에는 메뉴명과 추천 이유가 함께 표시된다.
3. **Given** 사용자가 추천 결과를 확인한 상태에서, **When** 다시 추천을
   요청하면, **Then** 서비스는 이전 결과와 다른 선택지를 우선 제시한다.

---

### User Story 2 - 내 상황에 맞는 추천 받기 (Priority: P2)

사용자는 먹고 싶은 음식의 종류, 예산, 이동 가능 거리, 식단 제한 같은 조건을
입력해 더 적합한 점심을 찾는다.

**Why this priority**: 기본 추천의 유용성을 높이지만, 조건 없는 추천이 먼저
동작해야 하므로 P2로 둔다.

**Independent Test**: 사용자가 하나 이상의 조건을 지정하고 추천을 요청하면,
결과가 지정 조건을 만족하는지 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 사용자가 음식 종류 또는 식단 제한을 지정한 상태에서, **When**
   추천을 요청하면, **Then** 결과는 해당 조건을 만족하는 메뉴만 포함한다.
2. **Given** 사용자가 예산을 지정한 상태에서, **When** 추천을 요청하면,
   **Then** 결과에는 예산을 초과하는 메뉴가 포함되지 않는다.
3. **Given** 지정한 조건을 만족하는 메뉴가 없는 상태에서, **When** 추천을
   요청하면, **Then** 서비스는 조건을 완화하거나 다시 선택할 수 있는 안내를
   제공한다.

### Edge Cases

- 조건을 전혀 입력하지 않아도 기본 추천이 제공되어야 한다.
- 가격, 영업 상태, 위치 정보가 없는 메뉴는 해당 조건을 적용한 결과에
  포함하지 않는다.
- 현재 이용 가능한 메뉴가 없으면 빈 화면 대신 재시도 또는 조건 완화 안내를
  제공한다.
- 사용자가 위치 권한을 거부하면 위치를 요구하는 실제 식당 추천 대신 주소나
  동네 입력을 안내한다.
- 사용자가 키보드만으로 조건 선택과 추천 요청을 완료할 수 있어야 한다.
- 추천 결과를 불러오는 중 오류가 발생하면 사용자가 다시 시도할 수 있는
  이해하기 쉬운 메시지를 표시한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 서비스는 사용자가 실제 식당 메뉴 또는 일반 메뉴 아이디어 중
  추천 유형을 먼저 선택할 수 있도록 해야 한다.
- **FR-002**: 서비스는 사용자가 별도 조건 없이 점심 추천을 요청할 수 있도록
  해야 한다.
- **FR-003**: 서비스는 실제 식당 메뉴 유형을 선택한 경우 주변 식당의 실제
  메뉴를, 일반 메뉴 아이디어 유형을 선택한 경우 위치와 무관한 메뉴 아이디어를
  추천해야 한다.
- **FR-004**: 서비스는 추천 결과에 메뉴명과 추천 이유를 표시해야 한다.
- **FR-005**: 실제 식당 메뉴 유형을 선택한 사용자는 위치 권한을 허용하거나
  주소·동네를 직접 입력할 수 있어야 한다.
- **FR-006**: 사용자는 음식 종류, 예산, 이동 가능 거리, 식단 제한 중 원하는
  조건을 선택하거나 입력할 수 있어야 한다.
- **FR-007**: 서비스는 사용자가 지정한 조건을 만족하지 않는 메뉴를 추천 결과에
  포함해서는 안 된다.
- **FR-008**: 사용자는 현재 추천 결과를 제외하고 다시 추천을 요청할 수 있어야
  한다.
- **FR-009**: 서비스는 조건을 만족하는 메뉴가 없을 때 원인과 다음 행동을
  안내해야 한다.
- **FR-010**: 서비스는 추천 요청 실패 시 재시도 방법을 제공해야 한다.
- **FR-011**: 서비스는 실제 식당 메뉴를 추천할 때 가격, 이용 가능 여부, 위치
  정보가 최신으로 확인된 항목만 결과에 포함해야 한다.
- **FR-012**: 서비스는 추천에 필요한 최소한의 사용자 입력만 수집하고, 저장이
  필요한 경우 사용자에게 그 목적을 알려야 한다.

### Key Entities *(include if feature involves data)*

- **점심 메뉴**: 실제 식당 메뉴 또는 일반 메뉴 아이디어. 실제 메뉴는 메뉴명,
  음식 종류, 가격, 위치, 이용 가능 여부, 식단 정보를 가질 수 있고, 모든
  결과는 추천에 활용되는 설명을 가진다.
- **추천 조건**: 음식 종류, 예산, 이동 가능 거리, 식단 제한 등 사용자가
  지정한 선호와 제약.
- **추천 결과**: 추천 조건에 따라 제시된 점심 메뉴와 각 메뉴의 추천 이유.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자의 90%가 추천 요청 후 10초 이내에 하나 이상의 결과를
  확인한다.
- **SC-002**: 사용자의 80%가 첫 방문에서 2분 이내에 추천 메뉴를 선택하거나
  다시 추천을 요청한다.
- **SC-003**: 조건을 지정한 추천 요청의 95%에서 모든 결과가 지정된 필수
  조건을 만족한다.
- **SC-004**: 추천 결과를 확인한 사용자의 70% 이상이 결과가 선택에 도움이
  되었다고 평가한다.
- **SC-005**: 추천 요청 실패 상황의 100%에서 사용자가 실패 원인과 재시도
  방법을 확인할 수 있다.
- **SC-006**: 실제 식당 메뉴 추천 결과의 100%가 추천 시점에 최신 정보 확인을
  완료한 항목으로 구성된다.

## Assumptions

- 서비스의 초기 대상은 점심을 고르는 개인 사용자다.
- 초기 버전은 식당 예약, 주문, 결제, 사용자 계정 관리를 제공하지 않는다.
- 메뉴 정보는 서비스가 신뢰할 수 있는 출처에서 제공받으며, 정보가 오래되었을
  가능성이 있음을 사용자에게 알릴 수 있다.
- 사용자가 위치를 제공하지 않으면 이동 가능 거리 조건은 적용하지 않는다.
- 추천은 사용자가 요청한 시점에 이용 가능한 점심 메뉴를 우선한다.
