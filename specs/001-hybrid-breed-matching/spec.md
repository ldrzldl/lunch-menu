# Feature Specification: 하이브리드 견종 매칭 대시보드

**Feature Branch**: `main` (별도 기능 브랜치 없음)

**Created**: 2026-07-15

**Status**: Draft

**Input**: 사용자 설명: 직접 제어와 접이식 설문 가이드를 결합하고, 설문 응답을
필터 값으로 매핑한 뒤 하드 필터링과 거리 기반 페널티로 Top 3 견종을 추천한다.

## Clarifications

### Session 2026-07-15

- Q: 설문 가이드는 어느 범위의 직접 제어 속성을 자동 설정해야 하는가? → A: 모든
  직접 제어 속성을 자동 설정한다.
- Q: 털 관련 직접 제어는 어떤 기준으로 구성해야 하는가? → A: `털 빠짐`과
  `미용·관리 부담`을 별도 범위 속성으로 제공한다.
- Q: 모든 속성의 설문 응답을 직접 제어 값으로 바꾸는 정책은 무엇인가? → A:
  범위형은 가중 점수, 범주형은 명시적 판정표를 사용한다.
- Q: 절대 조건만 활성화되어 거리 페널티를 계산할 수 없을 때 결과를 어떻게
  표시하는가? → A: 퍼센트 대신 `선택 조건 충족`으로 표시한다.
- Q: 범주형 설문 답변이 서로 충돌하면 어떻게 처리하는가? → A: 더 제한적인
  답변을 우선 적용한다.
- Q: 범위형 속성은 모두 후보를 배제하는가? → A: 크기와 짖음만 하드 필터로 쓰고,
  활동량·털 빠짐·미용은 후보를 유지한 채 거리 채점에만 사용한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 조건을 직접 조정해 후보 확인 (Priority: P1)

사용자는 대시보드에서 범위형 조건과 절대 조건을 직접 조정하고, 현재 조건을
통과하는 견종 수를 실시간으로 확인한 뒤 추천 결과를 요청한다.

**Why this priority**: 설문 없이도 명확한 선호를 바로 반영할 수 있는 핵심 경로이며,
필터링과 추천의 기반을 제공한다.

**Independent Test**: 모든 조건을 `상관없음`으로 시작해 활동량 범위, 주거 형태,
양육 경험을 차례로 변경하고 후보 수와 반환 결과가 각 조건과 일치하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 모든 조건이 `상관없음`인 초기 화면, **When** 사용자가 크기나 짖음
   범위를 지정하면, **Then** 지정 범위 안의 견종만 후보 수에 포함된다.
2. **Given** 범위 조건이 설정된 상태, **When** 사용자가 주거 형태나 양육 경험 같은
   절대 조건을 선택하면, **Then** 해당 조건을 충족하지 않는 견종이 즉시 후보에서
   제외되고 표시 수가 갱신된다.
3. **Given** 하나 이상의 유효 조건과 한 마리 이상의 후보가 있는 상태, **When**
   사용자가 결과를 요청하면, **Then** 시스템은 최대 3개의 결정적 추천 결과를
   순서대로 표시한다.

---

### User Story 2 - 설문으로 조건 자동 설정 (Priority: P2)

직접 값을 고르기 어려운 사용자는 숨겨진 설문 가이드를 펼쳐 모든 직접 제어 속성에
관한 생활 상황에 답하고, 시스템이 답변을 직접 제어 값으로 변환한 결과를 확인하고
필요하면 다시 조정한다.

**Why this priority**: 조건의 의미를 잘 모르는 사용자도 자신의 실제 생활을 통해
일관된 필터를 만들 수 있다.

**Independent Test**: 활동량 설문 두 문항과 주거·소음 설문 두 문항에 각각 낮음,
중간, 높음 조합으로 답하고 직접 제어 값이 정의된 매핑 결과로 바뀌는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 설문 가이드가 접힌 초기 화면, **When** 사용자가 가이드를 펼치면,
   **Then** 직접 제어 값은 유지되고 속성별 다중 문항이 나타난다.
2. **Given** 활동량 설문의 필수 문항을 모두 답한 상태, **When** 마지막 답변이
   선택되면, **Then** 가중 점수 구간에 맞는 활동량 범위가 직접 제어에 반영된다.
3. **Given** 주거·소음 설문 중 하나라도 최저 제약 답변이 포함된 상태, **When**
   매핑이 완료되면, **Then** 주거 조건은 공동주택으로, 짖음 허용 범위는 낮음으로
   설정된다.
4. **Given** 설문이 설정한 값을 사용자가 직접 변경한 상태, **When** 다른 조건을
   조작하면, **Then** 사용자가 마지막으로 직접 변경한 값이 현재 추천 조건으로
   유지된다.

---

### User Story 3 - 우선순위와 근거가 있는 결과 이해 (Priority: P3)

사용자는 가장 중요한 조건 하나를 지정하고, 추천 결과마다 일치율 또는 조건 충족
상태와 추천 이유, 확인할 점을 살펴보며 결과가 나온 이유를 이해한다.

**Why this priority**: 동일한 후보 안에서도 사용자의 핵심 선호를 반영하고 추천을
과도한 확신 없이 설명한다.

**Independent Test**: 같은 조건에서 우선순위만 변경해 순위와 페널티가 정의된 대로
달라지는지, 동일 입력을 반복했을 때 순서와 일치율이 같은지 확인한다.

**Acceptance Scenarios**:

1. **Given** 둘 이상의 활성 소프트 범위 조건, **When** 사용자가 한 조건을 1순위로
   선택하면, **Then** 해당 조건의 거리 페널티가 다른 조건의 2배로 반영된다.
2. **Given** 동일한 입력과 동일한 견종 데이터, **When** 추천을 반복 실행하면,
   **Then** 결과 순서와 일치율은 매번 동일하다.
3. **Given** 추천 결과가 반환된 상태, **When** 사용자가 결과를 확인하면, **Then**
   각 견종에 일치율 또는 조건 충족 상태, 주요 추천 이유, 주의할 점, 개체 차이에
   대한 안내가 표시된다.

### Edge Cases

- 모든 조건이 `상관없음`이면 후보 수는 전체 견종 수를 표시하되, 결과 요청 시
  최소 한 조건을 선택하도록 안내한다.
- 범위의 최솟값은 최댓값을 초과할 수 없으며, 같은 값은 유효한 단일값 범위다.
- 설문의 한 속성에 필요한 답변이 일부만 입력되면 해당 속성의 기존 직접 제어 값은
  바뀌지 않는다.
- 하드 필터 결과가 0마리이면 빈 결과와 함께 완화할 조건을 안내한다.
- 후보가 1~2마리이면 존재하는 후보만 반환하며 빈 순위를 채우지 않는다.
- 활성 절대 조건에 필요한 메타데이터가 없는 견종은 해당 조건을 충족하지 못한
  것으로 취급한다.
- 동일한 최종 페널티가 발생하면 안정적인 견종 식별자 순서로 순위를 결정한다.
- 설문을 접었다 다시 펼쳐도 현재 세션의 답변과 매핑된 직접 제어 값은 유지된다.
- 절대 조건만 활성화되면 후보 순서는 안정적인 견종 식별자 순서를 따르고 일치율
  대신 `선택 조건 충족`을 표시한다.
- 범주형 설문 답변이 충돌하면 판정표는 더 제한적인 값을 선택한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 연속·순서형 속성을 최소-최대 범위 제어로 제공해야 한다.
- **FR-002**: 시스템은 범주형 또는 절대 조건을 단일 선택 제어로 제공해야 한다.
- **FR-003**: 모든 직접 제어 속성은 해당 속성을 하드 필터 또는 소프트 채점에서 제외하는
  `상관없음` 선택을 제공해야 한다.
- **FR-004**: 초기 상태에서 모든 직접 제어는 `상관없음`이어야 한다.
- **FR-005**: 시스템은 직접 제어가 바뀔 때마다 하드 필터를 통과한 견종 수를
  200밀리초 이내에 갱신해야 한다.
- **FR-006**: 후보 수는 넓은 화면에서 제어 영역과 함께 계속 보여야 하며 작은
  화면에서도 현재 읽기 흐름 안에서 쉽게 확인할 수 있어야 한다.
- **FR-007**: 설문 가이드는 초기에는 접혀 있어야 하며 사용자가 필요할 때 펼치고
  다시 접을 수 있어야 한다.
- **FR-008**: 설문 가이드는 모든 직접 제어 속성을 자동 설정할 수 있어야 하며,
  각 속성마다 서로 다른 생활 신호를 묻는 문항을 최소 2개 포함해야 한다.
- **FR-009**: 한 속성의 필수 설문 답변이 모두 입력되면 시스템은 정의된 매핑 규칙에
  따라 대응하는 직접 제어 값을 자동으로 갱신해야 한다.
- **FR-010**: 설문 매핑 후에도 사용자는 직접 제어 값을 수정할 수 있어야 하며,
  설문 또는 직접 제어 중 마지막으로 변경된 입력이 현재 값이 되어야 한다.
- **FR-011**: 활동량 설문 점수는 산책 가능 시간 60%, 휴일 활동 스타일 40%로
  계산해야 한다. 가중 점수가 1.0 이상 1.7 미만이면 범위 1~2, 1.7 이상 2.4
  미만이면 2~4, 2.4 이상이면 3~5로 설정해야 한다.
- **FR-012**: 주거·소음 설문에서 거주 공간 또는 방음 상태 답변 중 하나라도 최저
  제약값이면 주거 조건을 `공동주택`, 짖음 허용 범위를 1~2로 설정해야 한다. 그 외
  경우 주거 조건은 실제 거주 공간 답변을 따르고, 방음 상태가 보통이면 짖음 범위를
  1~3, 양호하면 `상관없음`으로 설정해야 한다.
- **FR-013**: 하드 필터는 활성 절대 조건과 활성 크기·짖음 범위의 양 끝값을 포함해
  모두 충족하는 견종만 남겨야 한다. 활동량·털 빠짐·미용 범위는 후보를 배제하지 않는다.
- **FR-014**: `상관없음`인 속성은 하드 필터와 페널티 계산에 영향을 주지 않아야 한다.
- **FR-015**: 초보 양육 조건이 활성화되면 초보 친화 조건을 충족하지 못하는 견종은
  후보에서 제외되어야 한다.
- **FR-016**: 어린아이 동거 조건이 활성화되면 해당 환경 적합 조건을 충족하지
  못하는 견종은 후보에서 제외되어야 한다.
- **FR-017**: 소프트 채점은 활성 활동량·털 빠짐·미용 범위의 중앙값과 견종 속성값
  사이 절대 거리의 합을 페널티로 사용해야 한다.
- **FR-018**: 사용자는 활성 소프트 범위 속성 중 하나 또는 `없음`을 1순위 조건으로 선택할
  수 있어야 하며, 선택된 속성의 거리 페널티에는 2.0 가중치가 적용되어야 한다.
- **FR-019**: 시스템은 총 페널티가 낮은 순서로 후보를 정렬하고, 동점이면 안정적인
  견종 식별자 오름차순으로 정렬해야 한다.
- **FR-020**: 시스템은 최대 3개 결과를 반환하고, 활성 조건에서 가능한 최대
  페널티 대비 해당 견종의 페널티 비율을 `100 × (1 - 총 페널티 ÷ 최대 페널티)`로
  환산해 반올림한 0~100 일치율을 표시해야 한다. 각 속성의 최대 페널티는 범위
  중앙값에서 해당 속성 척도의 가장 먼 끝값까지의 거리이며 우선순위 가중치를
  포함한다. 페널티 0은 일치율 100%이며 값은 범위를 벗어나지 않아야 한다. 활성
  소프트 범위가 없고 절대 조건만 있으면 일치율을 계산하지 않고 `선택 조건 충족`으로
  표시해야 한다.
- **FR-021**: 각 결과는 일치율 또는 조건 충족 상태, 주요 추천 이유, 가장 주의가
  필요한 조건, 개체별 성격과 건강 차이에 대한 안내를 포함해야 한다. 주요 추천
  이유는 활성 소프트 범위가 있으면 가장 가까운 범위 조건, 없으면 충족한 절대 조건에서
  선택해야 한다.
- **FR-022**: 시스템은 초기화 동작을 제공하고 직접 제어, 설문 답변, 우선순위,
  후보 수, 결과를 모두 초기 상태로 되돌려야 한다.
- **FR-023**: 모든 직접 제어가 `상관없음`이면 시스템은 전체 후보 수를 보여주되
  결과를 반환하지 않고 하나 이상의 조건을 선택하도록 한국어로 안내해야 한다.
- **FR-024**: 시스템은 `털 빠짐`과 `미용·관리 부담`을 서로 독립적인 소프트 범위
  조건으로 제공하고 페널티에서 각각 계산해야 한다.
- **FR-025**: 범위형 속성의 설문 매핑은 문항별 가중치의 합이 100%인 가중 점수를
  낮음·중간·높음 구간으로 나눠 직접 범위로 변환해야 한다. 범주형 속성은 가능한
  모든 완성 답변 조합을 하나의 허용된 단일 선택 값으로 연결하는 명시적 판정표를
  사용해야 한다.
- **FR-026**: 범주형 설문 답변이 충돌하면 판정표는 더 제한적인 값을 선택해야 한다.
  제한 순서는 주거 조건에서 `공동주택`, 양육 경험에서 `초보`, 어린아이 동거에서
  `예`가 각각 반대 값보다 우선한다.

### Constitution Requirements *(mandatory when applicable)*

- **Responsible Guidance**: 추천은 정보 제공용이며 입양 적합성 판정이나 품종의
  우열로 표현하지 않는다. 모든 결과에 근거, 주의점, 개체 차이 안내를 제공한다.
- **Accessibility**: 모든 제어, 설문 펼침, 초기화, 결과 요청은 키보드로 사용할 수
  있어야 한다. 제어에는 한국어 이름과 현재 값이 제공되고, 후보 수 갱신은 포커스를
  이동시키지 않으며 보조기술에도 전달되어야 한다.
- **Privacy**: 직접 선택과 설문 답변은 현재 브라우저 세션 안에서만 사용하고 저장,
  전송, 분석 추적을 하지 않는다.
- **Verification**: 모든 범위·범주형 매핑, 범주형 충돌, Hard/Soft 분리, 우선순위 가중치,
  동점 순서, 일치율 경계와 절대 조건 전용 상태를 하나의 결정적 회귀 검사로
  검증하고 전체 직접·설문 경로를 확인한다.

### Key Entities *(include if feature involves data)*

- **견종**: 안정적인 식별자, 표시 이름, 활동량·털 빠짐·미용 관리·짖음 등 범위
  채점용 1~5 속성, 1~4 크기 단계, 주거 적합성, 초보 친화 여부, 어린아이 동거
  적합성, 설명과 주의 정보를 가진다.
- **필터 구성**: 각 범위의 최소·최대값 또는 `상관없음`, 절대 조건 값, 1순위 조건을
  나타내는 현재 추천 입력이다.
- **설문 응답**: 속성별 문항 선택과 응답 시점을 가지며 매핑 규칙의 입력이 된다.
- **매핑 규칙**: 범위형에는 문항별 가중치, 완성 조건, 점수 구간과 직접 범위를,
  범주형에는 모든 완성 답변 조합, 제한 순서와 대응하는 단일 선택 값을 정의한다.
- **추천 결과**: 견종, 총 페널티, 일치율 또는 조건 충족 상태, 추천 이유, 확인할 점,
  최종 순위를 가진다.

## Scope Boundaries

- 이번 기능은 현재 제공되는 40종 견종 데이터와 브라우저 내 추천 흐름만 다룬다.
- 계정, 서버 저장, 분석 추적, 원격 추천 서비스, 기계학습 예측은 포함하지 않는다.
- 입양 승인, 의료·행동 전문가 판단, 개별 동물의 건강 예측은 포함하지 않는다.
- 견종 데이터 관리 화면과 사용자별 장기 기록은 포함하지 않는다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 도움 없이 1분 안에 직접 조건을 설정하고 후보 수를 확인할
  수 있다.
- **SC-002**: 모든 직접 제어 변경에서 후보 수의 95% 이상이 200밀리초 이내에
  갱신된다.
- **SC-003**: 모든 직접 제어 속성에 대해 정의된 설문 조합의 100%가 명세된 직접
  제어 값으로 매핑된다.
- **SC-004**: 동일한 입력과 데이터로 100회 반복한 결과의 순서와 일치율 또는 조건
  충족 상태가 모두 같다.
- **SC-005**: 키보드만 사용하는 사용자가 직접 설정, 설문 설정, 결과 확인, 초기화를
  모두 완료할 수 있다.
- **SC-006**: 반환된 추천 결과의 100%가 일치율 또는 조건 충족 상태, 추천 이유,
  확인할 점, 개체 차이 안내를 포함한다.
- **SC-007**: 0개 후보와 3개 미만 후보를 포함한 경계 시나리오에서 잘못된 빈 카드나
  존재하지 않는 순위가 표시되지 않는다.

## Assumptions

- 초기 Top-K 값은 현재 제품과 같은 3으로 고정한다.
- 범위형 속성은 활동량, 털 빠짐, 미용·관리 부담, 짖음, 크기를 대상으로 하며 값이
  클수록 해당 속성의 정도가 높다.
- 절대 조건은 주거 형태, 양육 경험, 어린아이 동거 여부를 우선 대상으로 한다.
- 실시간 후보 수는 소프트 채점 전 하드 필터를 통과한 견종 수를 의미한다.
- 설문이 값을 갱신한 뒤 직접 제어를 수정하면 수동 값이 우선하며, 관련 설문 답변을
  다시 바꾸면 새 매핑값이 우선한다.
- 사용자가 요구한 동적 후보 수는 추천 결과를 미리 계산해 노출하는 기능이 아니라
  현재 조건의 폭을 이해하도록 돕는 피드백이다.
