# Feature Specification: 강아지 양육 시뮬레이션

**Feature Branch**: `001-puppy-care-simulation`

**Created**: 2026-07-13

**Status**: Draft

**Input**: User description: "반려동물을 키우기 전에는 양육 시 필요한 책임감을 실감하기 어렵다. 강아지를 키우고 싶은 사람이 실제로 감당할 수 있을지 미리 확인하도록 불규칙한 시간 알림과 산책·밥·배변 처리 등의 케어 행동을 시뮬레이션한다. 실제 반려동물 매칭·분양은 제공하지 않는다."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 강아지 돌봄 일정 체험하기 (Priority: P1)

강아지를 키운 적 없는 사용자는 시뮬레이션을 시작하고, 예측하기 어려운
시간에 발생하는 돌봄 요청을 받아 실제 양육에서 요구되는 시간적 책임을
체험한다.

**Why this priority**: 사용자가 양육 책임을 직접 경험하는 것이 문제를 해결하는
핵심 가치이므로 가장 먼저 제공해야 한다.

**Independent Test**: 사용자가 시뮬레이션을 시작한 뒤 정해진 체험 기간 동안
돌봄 요청을 받고 처리 여부를 기록할 수 있는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 시뮬레이션을 시작한 상태에서, **When** 돌봄 요청 시간이
   되면, **Then** 사용자는 요청 유형과 처리 기한을 알림으로 확인한다.
2. **Given** 돌봄 요청을 확인한 상태에서, **When** 사용자가 요청을 열면,
   **Then** 해당 행동을 완료하거나 나중에 처리하도록 선택할 수 있다.
3. **Given** 사용자가 처리 기한이 지난 요청을 가진 상태에서, **When** 요청
   상태를 확인하면, **Then** 지연 사실과 누적된 돌봄 부담을 확인한다.

---

### User Story 2 - 돌봄 행동 수행하기 (Priority: P1)

사용자는 산책, 밥 주기, 물 갈기, 배변 처리, 상태 확인 등 강아지에게 필요한
행동을 수행하고, 각 행동에 필요한 시간과 반복 부담을 경험한다.

**Why this priority**: 알림을 받는 것만으로는 책임감을 체감하기 어려우므로
실제 행동 수행과 누적 부담 확인이 핵심 체험에 포함되어야 한다.

**Independent Test**: 각 돌봄 유형을 하나 이상 수행하고, 완료 기록과 소요
시간이 시뮬레이션 결과에 반영되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 산책 요청이 표시된 상태에서, **When** 사용자가 산책 완료를
   기록하면, **Then** 요청은 완료 상태가 되고 해당 돌봄 기록에 반영된다.
2. **Given** 사용자가 여러 돌봄 요청을 처리한 상태에서, **When** 체험 현황을
   확인하면, **Then** 유형별 완료율, 지연 횟수, 사용 시간, 남은 요청을 볼 수
   있다.
3. **Given** 사용자가 요청을 처리하기 어려운 상태에서, **When** 나중에 처리를
   선택하면, **Then** 요청은 미완료 상태로 남고 이후 부담에 반영된다.

---

### User Story 3 - 양육 준비도 돌아보기 (Priority: P2)

사용자는 체험이 끝난 뒤 자신의 돌봄 수행 패턴과 어려웠던 상황을 돌아보고,
실제 입양을 결정하기 전에 자신의 시간·생활 여건을 점검한다.

**Why this priority**: 최종 목적은 게임 점수가 아니라 실제 양육을 감당할 수
있는지 판단하도록 돕는 것이지만, 돌봄 체험이 먼저 제공되어야 한다.

**Independent Test**: 체험 종료 후 사용자가 요약 결과와 자기 점검 질문을
확인하고, 입양 전 고려사항을 기록할 수 있는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 체험 기간이 종료된 상태에서, **When** 사용자가 결과를 열면,
   **Then** 돌봄 유형별 수행 현황과 반복적으로 놓친 요청을 확인한다.
2. **Given** 사용자가 결과를 확인한 상태에서, **When** 자기 점검 질문에
   응답하면, **Then** 응답 내용과 체험 결과를 바탕으로 준비 전 확인할
   생활 조건을 안내받는다.
3. **Given** 사용자가 체험 결과를 확인한 상태에서, **When** 실제 입양 관련
   행동을 선택하면, **Then** 서비스는 매칭·분양을 수행하지 않고 신뢰할 수
   있는 사전 조사 또는 상담을 권한다.

### Edge Cases

- 사용자가 알림 권한을 허용하지 않거나 기기 알림을 놓쳐도 다음 접속 시
  미확인 요청과 처리 기한을 확인할 수 있어야 한다.
- 사용자가 체험 중 일부 요청을 연속으로 놓쳐도 시뮬레이션은 종료되지 않고
  누적 부담과 지연 기록을 보여줘야 한다.
- 사용자가 알림을 받은 뒤 실제 행동을 수행할 수 없는 시간대여도 요청을
  완료, 나중에 처리, 미완료 중 하나로 기록할 수 있어야 한다.
- 사용자가 체험을 중단했다가 다시 시작하면 기존 기록을 이어 보거나 초기화할
  수 있어야 한다.
- 사용자가 접근성 기능을 사용하는 경우에도 알림 내용, 요청 상태, 완료
  조작, 결과 요약을 파악할 수 있어야 한다.
- 알림 또는 기록 저장에 실패하면 사용자는 실패 사실과 재시도 방법을
  이해할 수 있어야 하며, 이미 완료한 기록이 중복 처리되어서는 안 된다.
- 시뮬레이션 결과가 실제 강아지의 개별 행동이나 양육 난이도를 정확히
  예측한다고 오해하지 않도록 서비스의 한계를 안내해야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 서비스는 사용자가 강아지 양육 시뮬레이션을 시작, 일시 중단,
  재개, 종료할 수 있도록 해야 한다.
- **FR-002**: 서비스는 사용자의 예측을 어렵게 하는 시간 간격으로 돌봄 요청을
  생성하고, 요청 유형과 처리 기한을 알려야 한다.
- **FR-003**: 돌봄 요청 유형에는 산책, 밥 주기, 물 갈기, 배변 처리, 상태
  확인이 포함되어야 한다.
- **FR-004**: 사용자는 각 돌봄 요청을 완료, 나중에 처리, 미완료 중 하나로
  기록할 수 있어야 한다.
- **FR-005**: 서비스는 돌봄 요청별 발생 시각, 상태 변경 시각, 요청 유형을
  기록하고 중복 완료 기록을 방지해야 한다.
- **FR-006**: 서비스는 사용자가 체험 중 현재 미완료 요청, 지연 횟수, 유형별
  완료 현황, 누적 돌봄 시간을 확인할 수 있도록 해야 한다.
- **FR-007**: 서비스는 체험 종료 후 돌봄 유형별 수행 현황, 놓친 요청, 반복된
  어려움을 요약해야 한다.
- **FR-008**: 서비스는 결과 요약과 함께 실제 양육 전에 점검할 시간, 생활환경,
  비용, 가족 동의, 장기 책임에 관한 자기 점검 질문을 제공해야 한다.
- **FR-009**: 서비스는 알림 권한이 없거나 알림을 놓친 사용자에게 다음 접속
  시 미확인 요청과 처리 상태를 제공해야 한다.
- **FR-010**: 서비스는 돌봄 기록 저장 실패 시 사용자에게 재시도 방법을
  제공하고, 저장되지 않은 기록을 완료된 것으로 표시해서는 안 된다.
- **FR-011**: 서비스는 키보드 조작과 보조기술 사용자가 돌봄 요청, 상태,
  결과 요약을 이해하고 조작할 수 있도록 해야 한다.
- **FR-012**: 서비스는 시뮬레이션 결과가 실제 반려견의 행동·난이도·입양
  적합성을 보장하지 않는다는 안내를 제공해야 한다.
- **FR-013**: 서비스는 실제 반려동물 매칭, 분양, 입양 신청 또는 판매를
  수행해서는 안 된다.
- **FR-014**: 서비스는 시뮬레이션에 필요한 정보만 수집하고, 개인 생활 패턴과
  돌봄 기록의 수집 목적과 보관 여부를 사용자에게 알려야 한다.

### Key Entities *(include if feature involves data)*

- **시뮬레이션 세션**: 사용자의 한 번의 강아지 양육 체험. 시작·중단·재개·종료
  상태, 체험 기간, 현재 현황을 가진다.
- **돌봄 요청**: 특정 시점에 발생한 산책·급식·물·배변·상태 확인 행동. 유형,
  발생 시각, 처리 기한, 상태를 가진다.
- **돌봄 기록**: 사용자의 돌봄 요청 처리 결과. 처리 상태와 상태 변경 시각,
  소요 시간을 가진다.
- **준비도 회고**: 체험 결과 요약과 사용자의 자기 점검 응답. 실제 입양
  적합성을 판정하지 않고 준비 전 고려사항을 확인하는 데 사용한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 신규 사용자의 80% 이상이 시뮬레이션 시작 후 첫 돌봄 요청을
  확인하고 5분 이내에 처리 상태를 기록한다.
- **SC-002**: 시뮬레이션을 끝까지 완료한 사용자의 80% 이상이 결과 요약에서
  자신의 돌봄 수행 현황과 놓친 요청을 확인한다.
- **SC-003**: 돌봄 요청 처리 기록의 99% 이상이 중복 완료 없이 정확한 상태로
  보존된다.
- **SC-004**: 알림 권한을 허용하지 않은 사용자의 95% 이상이 다음 접속 시
  미확인 돌봄 요청을 확인할 수 있다.
- **SC-005**: 체험 완료 사용자의 70% 이상이 결과 확인 후 실제 양육 전에
  점검할 생활 조건을 하나 이상 기록하거나 확인한다.
- **SC-006**: 서비스 이용 후 6개월 이내 실제 입양한 사용자 집단의 파양 또는
  양육 포기 비율이 사전에 정의한 유사 대조군보다 낮은지 추적·비교할 수
  있어야 한다. 이 지표는 장기 효과 검증 대상으로 사용하며, 서비스 단독의
  인과 효과를 보장하는 기준으로 해석하지 않는다.
- **SC-007**: 사용성 평가에서 참여자의 90% 이상이 알림 확인, 돌봄 상태 기록,
  결과 요약 확인의 세 가지 핵심 흐름을 별도 도움 없이 완료한다.

## Assumptions

- 초기 대상은 반려견을 키운 경험이 없고 실제 입양을 고려하는 개인 사용자다.
- 체험은 실제 동물을 대신하거나 입양 여부를 자동 판정하지 않는 사전 점검
  도구다.
- 기본 체험 기간과 돌봄 요청 빈도는 사용자가 부담을 이해할 수 있을 만큼의
  짧은 기간으로 제공하며, 구체적인 값은 계획 단계에서 정한다.
- 사용자는 요청을 처리할 수 없는 상황이 있을 수 있으므로 미완료 기록도
  정상적인 결과로 취급한다.
- 장기 파양·포기 비교를 위해서는 사용자의 자발적 후속 조사 참여와 실제
  입양 여부 확인이 필요하며, 서비스는 이를 위해 필요한 최소 정보만 수집한다.
- 실제 입양 관련 정보는 사용자가 별도로 확인해야 하며, 서비스는 매칭·분양
  기관의 대리 역할을 하지 않는다.
